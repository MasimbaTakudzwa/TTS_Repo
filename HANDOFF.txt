1. WHAT THIS IS
-------------------------------------------------------------------
  - A local, offline text-to-speech pipeline with VOICE CLONING.
  - It turns a text string, a .txt/.md file, or a PDF into a spoken
    WAV file, optionally in a cloned voice built from a short
    reference recording (6-30s of clean speech).
  - It is a LOCAL MODEL pipeline (NOT an API). The engine is Coqui
    XTTS v2 (the maintained "coqui-tts" fork). No API keys, no usage
    limits, no internet required after the model is downloaded once.
  - GOAL: a personal tool to read documents/notes aloud in Masimba's
    own cloned voice. The repo also sketches a paid ElevenLabs API
    alternative, but that path was never built (see section 6).
  - Maturity: working personal-use script / hobby tool. It runs and
    produces audio. It is NOT a packaged product, has no automated
    tests, and the "test*.py" files are ad-hoc smoke scripts.

2. TECH STACK & ARCHITECTURE

  Language: Python (developed against Python 3.11; see venv note below)

  Core libraries / models (verified against the working venv):
    - coqui-tts (PyPI package name "TTS") 0.22.0  -> XTTS v2 model
    - torch 2.5.1, torchaudio 2.5.1, torchvision 0.20.1

    - soundfile + numpy    -> WAV merge / crossfade / peak-normalize

  Model used: tts_models/multilingual/multi-dataset/xtts_v2

3. CURRENT STATE (honest)
-------------------------------------------------------------------
  WORKS:
    - End-to-end synthesis via the free local pipeline: text/file/PDF
      -> WAV, with voice cloning from a reference clip.
    - CLI (main_free.py) and module API (TTSPipeline) both function.
    - ffmpeg reference preprocessing (mono, 24kHz resample, silence
      trim, EBU R128 loudnorm) with a plain-conversion fallback.
    - Text preprocessing (abbreviations, long numbers via inflect,
      URL/email stripping, dash->comma, punctuation cleanup).
    - Word-based chunking (~180 words) with crossfaded WAV merge and
      final peak normalization.
    - Device auto-detect. In practice this runs on CPU on the Mac:
      the code DELIBERATELY forces CPU even when MPS is detected
      because XTTS inference on Apple MPS is broken/unreliable
      (see _load_model in tts_pipeline_free.py). CPU is slow for long
      text. The 3060 RTX (CUDA) laptop is the fast path.

      speed=1.09). These call-site overrides differ from the class

7. HOW TO RUN & DEVELOP
-------------------------------------------------------------------
  Prereqs: Python 3.11 (NOT 3.13), ffmpeg on PATH, internet for the
  one-time ~1.8 GB model download.

  Recommended: recreate the venv fresh (do NOT rely on the copied
  venvs - their internal paths are stale, see section 6).

    python3.11 -m venv tts-venv-311
    source tts-venv-311/bin/activate
    pip install --upgrade pip
    pip install coqui-tts pdfplumber inflect soundfile numpy

  (Pin to the known-good set if needed: coqui-tts==0.22.0,
   torch==2.5.1, torchaudio==2.5.1. coqui-tts pulls torch in.)

  Run the CLI:
    python main_free.py --text "Hello, this is my cloned voice." --ref my_voice.wav
    python main_free.py --file notes.txt --ref my_voice.wav --output notes.wav
    python main_free.py --file document.pdf
    python main_free.py --list-langs
    python main_free.py --text "Hello" --cpu        (force CPU)

  Run the active experiment:
    python read.py            (clones lossless.mp4 -> cloned.wav)

  Use as a module:
    from tts_pipeline_free import TTSPipeline
    pipe = TTSPipeline(reference_audio="my_voice.wav")
    pipe.speak_text("Hello world", output_path="out.wav")

  CRITICAL - large assets NOT on GitHub (drive-only):
    A GitHub clone is INCOMPLETE. It does NOT include the virtual
    environments (the multi-GB pieces). You must either:
      (a) RECREATE the 3.11 venv with the pip commands above
          (recommended - cleaner than copying stale-path venvs), OR
      (b) copy them from the drive (see section 9), accepting that
          their internal paths point at the old ~/Desktop location.
    The XTTS model (~1.8 GB) is also not on GitHub and not on the
    drive as files - it re-downloads on first run.
    The reference audio my_voice.wav (best/largest, 7.3MB) is
    gitignored and lives on the drive; lossless.mp4 and my_voice.mpeg
    DID get tracked and ARE on GitHub.

  Env vars / API keys:
    - NONE required for the free local pipeline (it is fully offline).
    - If the unbuilt ElevenLabs path is ever implemented, it would
      need an ELEVENLABS_API_KEY env var. There is no such key today.
